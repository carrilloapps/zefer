# app/lib — Core Logic

All encryption, compression, device detection, and data handling logic. No UI, no React.

## Files

| File | Purpose | Key Exports |
|---|---|---|
| `crypto.ts` | AES-256-GCM encryption/decryption, PBKDF2 key derivation, benchmarking | `encrypt()`, `decrypt()`, `encryptBytes()`, `decryptBytes()`, `combineDualKeys()`, `hashAnswer()`, `benchmarkDevice()` |
| `zefer.ts` | ZEFER3 file format encode/decode, all security checks | `encodeZefer()`, `decodeZefer()`, `parseFile()` |
| `compression.ts` | Gzip/Deflate via browser CompressionStream API | `compress()`, `decompress()` |
| `device.ts` | RAM/CPU/GPU detection, dynamic file size limits | `analyzeDevice()`, `formatBytes()` |
| `i18n.ts` | ~200 translation keys in es/en/pt | `t()`, `LOCALES`, `Locale`, `TranslationKey` |
| `ip.ts` | Public IP detection via ipify.org, IP allowlist check | `getClientIp()`, `parseIpList()`, `isIpAllowed()` |
| `preferences.ts` | Persisted user preferences (TTL) via localStorage | `usePreferences()` |
| `progress.ts` | Encryption/decryption progress tracking with device-calibrated timing | `createEncryptTracker()`, `createDecryptTracker()` |

## Critical Patterns

- `crypto.ts` uses `Uint32Array` for uniform randomness in key generation
- `zefer.ts` payload format: `metadata_json + \0 + raw_bytes` — single AES-GCM block
- `device.ts` uses `performance.memory.jsHeapSizeLimit` (Chrome) → `navigator.deviceMemory` → cores heuristic (fallback chain)
- `i18n.ts` keys are typed — `TranslationKey` is a union of all literal key strings
- All functions are pure/async — no React dependencies
